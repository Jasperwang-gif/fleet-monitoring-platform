今天目标：

Day 4：Telemetry 模块 API

今天目标：

让系统能接收车辆上传的数据

你今天要完成：

写 TelemetryRecord 模块
做这些 API：
POST /telemetry
GET /vehicles/{id}/telemetry/latest
GET /vehicles/{id}/telemetry/history

你今天要注意：

POST /telemetry 要校验 vehicle 是否存在
history 查询最好支持时间倒序
latest 查询可以只取最近一条

今天产出：

每辆车都能看到最新状态和历史记录
Day 5：Alert 告警逻辑

今天目标：

做出“有业务味道”的核心功能

你今天要完成：

写 Alert 模块
做这些 API：
GET /alerts
GET /vehicles/{id}/alerts
PATCH /alerts/{id}/resolve

然后加业务规则：

速度 > 100，生成 OVER_SPEED
电量 < 20，生成 LOW_BATTERY

如果你还想再加一个：

超过一定时间没数据，视为 OFFLINE

今天产出：

telemetry 上传时，系统自动判断是否生成 alert

这一天很关键，因为它让你的项目从“数据展示”变成“业务系统”。

六、简历上怎么写这个项目

你以后可以写成这样：

Fleet Monitoring Platform
Built a full-stack fleet monitoring system using Java Spring Boot, React, and PostgreSQL to track vehicle telemetry, generate alerts, and visualize fleet status in a dashboard.

你再拆成 3 条：

Designed and implemented RESTful APIs for vehicle management, telemetry ingestion, and alert handling using Spring Boot and PostgreSQL.
Built business rules to automatically generate alerts for overspeed and low-battery events based on incoming telemetry data.
Deployed the full-stack application to cloud infrastructure and documented the system with Swagger, README, and test coverage.

这几句很像真实工程项目，不像学生作业。

七、面试时你要会讲的 6 个问题

你一定要提前准备这几个：

1. 为什么做这个项目？

回答方向：

想做一个更接近真实企业场景的系统
对 IoT / fleet / telemetry / backend system 感兴趣
本科机器人背景和软件方向结合
2. 最难的地方是什么？

回答方向：

telemetry 和 alert 的数据关系
设计 latest/history 查询
鉴权和错误处理
3. 为什么这样设计数据库？

回答方向：

vehicle 是主体
telemetry 是高频时序数据
alert 是业务事件
这样方便扩展和查询
4. 你怎么保证代码质量？

回答方向：

DTO
service/controller 分层
全局异常处理
单元测试
Git 规范
5. 如果要扩展，你会怎么做？

回答方向：

WebSocket 实时推送
地图展示
Kafka 处理高频数据
Redis 缓存 dashboard
6. 你在这个项目里学到了什么？

回答方向：

API design
schema design
debugging
deployment
business-oriented engineering thinking
